Original Super Mario Bros Clone - CSE 3902

Language(s): C#

Tools: Visual Studio 2017, MonoGame, Visual Studio Team Services

Project Description

Created a clone of the first level of Super Mario Bros in teams of four, using Agile Scrum methodologies. The project introduced fundamental concepts to writing readable and maintainable code. The project was divided into four sprints, each lasting about two weeks. I have to keep the repository private, but feel free to contact me and, I can share the code with you.

Sprint Details*

Sprint 1

  • Create interfaces for different types of 2D platforming game elements/objects (examples could include: players, enemies, items, projectiles, platforms, blocks)
  • Create a class for each object included in the first level of your game.
  • Implement the animation (update + draw) for the default state of each object.
  • Implement the animation (update + draw) for all possible states of the player character that can be reached via user input (i.e. pressing left/right/up/down, run, and jump buttons) and by gaining items, except for temporary powers (i.e. star in Mario Bros.)

Sprint 2

  • Implement up/down/left/right movement for Mario. It should not be physically based / jumping based. Move Mario left when left is pushed, up when up is pushed, etc. The visual effect is the same motion for a top-down 2D game such as the original Legend of Zelda. Diagonal movement is possible when pushing multiple keys at the same time.
  • Implement state transitions, collision detection, and collision response for Mario vs. enemies, blocks, and items. Mario should not be able to pass through blocks and any state changes for objects occur instantly (i.e. you do not have to animate between states after Mario gains an item or takes damage).
  • Add background elements (clouds and bushes) as individual objects or as part of one background image.
  • Define a level via a csv or xml file and write code to initialize objects based on the file's contents.
  • Create an artificial level that contains an instance of Mario and all types of blocks, pipes, powerups/items, and enemies that are found in level 1-1, but place them such that they all fit on the screen at the same time. The image below displays one possible arrangement. For this sprint, inclusion of the flagpole and castle are optional.

Sprint 3

  • Implement the core gameplay of a 2D platforming game.
  • Update your level file to include all of the objects in the first level of your game. Additionally, at the start of the level add an instance of every block and item type. That way you (and the grader) will save time during manual testing.
  • Implement movement and jumping for the player character (Mario).
  • Implement the player's transitional states (changing powerups or taking damage).
  • Implement movement for enemies.
  • Implement movement for powerups and items.
  • Add any projectiles made by players or enemies (Mario's fireballs).
  • Add movement for blocks and platforms (where applicable, such as block bumping).
  • Update collision detection and handling to deal with the new types of moving objects. You may also need to make modifications to handle cases where objects are colliding with two other objects at the same time (ex: mario intersecting with two blocks)
  • Add a scrolling camera (you may or may not enforce the restriction of never scrolling back to the left - better yet use an interface and implement both!).
  • Implement resetting the level upon the player character's death.

Sprint 4

  • Add the Heads Up Display.
  • Add sounds.
  • Add points and the score system
  • Add coins, the coin system, and lives.
  • Implement game state transitions:
    • The remaining lives screen before/after deaths
    • Warp pipe transition to secret underground room (and coming back up again)
    • Pausing
    • The winning game state and flagpole animation when Mario reaches the end of the level
  • Implement any other features you deem necessary for copying gameplay on level 1-1.
